# torch is installed from a backend-specific wheel index (CPU/CUDA/ROCm) by the
# Dockerfiles / run-native.py, pinned to the 2.9 series; this line is stripped

# Tighter lower bounds here are deliberate. With wide-open floors, pip's resolver
# has to consider years of releases of transformers / tokenizers / huggingface_hub
# / safetensors (which constrain each other) on top of the pinned ROCm/CUDA torch
# build -- whose +rocm local version isn't on any index, so pip can't prune the
# torch axis from candidates either -- and it backtracks until it gives up with
# "resolution-too-deep". Raising the floors (and pinning the deep transitive HF
# packages tokenizers / safetensors explicitly) to recent, mutually-compatible
# versions caps the backtracking depth so it resolves. These floors still admit
# the exact versions pip picks today (transformers 5.x, tokenizers 0.22.x).

accelerate>=1.0.0
scipy>=1.13.0
huggingface_hub>=0.34.0
# tokenizers / safetensors are pulled in by transformers with loose pins and have
# many releases; explicit floors keep the resolver from walking their history.
